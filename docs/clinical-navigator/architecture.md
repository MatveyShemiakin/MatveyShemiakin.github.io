# Архитектура клинического навигатора

## 1. Разделение ответственности

### Frontend

Отвечает за:

- интерфейс врача;
- пошаговый сбор обезличенных данных;
- отображение предупреждений и результатов;
- локальную проверку обязательных полей;
- передачу запроса на backend.

Frontend не хранит секретные ключи, не принимает окончательных клинических решений и не содержит производственную лекарственную базу.

### Backend

Отвечает за:

- авторизацию и проверку профессионального доступа;
- очистку и структурирование входных данных;
- вызов языковой модели через API;
- поиск по утверждённой базе знаний;
- выполнение детерминированных клинических правил;
- проверку красных флагов;
- блокировку небезопасного ответа;
- журналирование версии алгоритма и источников;
- лимиты, аудит и мониторинг ошибок.

### Языковая модель

Разрешённые функции:

- понимать свободный медицинский текст;
- извлекать клинические параметры;
- задавать уточняющие вопросы;
- объяснять результат;
- формировать структурированный отчёт.

Запрещённые функции:

- придумывать препараты, дозы или длительность;
- использовать открытый интернет в клиническом режиме;
- игнорировать вывод rule engine;
- формировать лечение при недостаточных данных;
- изменять утверждённую базу знаний.

### Rule engine

Детерминированный модуль должен определять:

- обязательные вопросы;
- красные флаги;
- условия остановки сценария;
- диапазоны допустимых значений;
- правила фенотипизации;
- противопоказания и блокировки;
- необходимость срочной маршрутизации.

### База знаний

Каждая запись должна содержать:

- уникальный идентификатор;
- тип источника;
- название документа;
- организацию и юрисдикцию;
- год и версию;
- конкретный раздел или таблицу;
- клиническое утверждение;
- уровень уверенности;
- дату экспертной проверки;
- статус: draft / approved / deprecated.

## 2. Предлагаемый поток запроса

```text
Врач
  ↓
Frontend сайта
  ↓
Backend API
  ├─ проверка персональных данных
  ├─ проверка обязательных полей
  ├─ rule engine: красные флаги
  ├─ поиск по утверждённой базе знаний
  ├─ вызов языковой модели
  ├─ проверка структуры ответа
  └─ safety validator
  ↓
Структурированный результат врачу
```

## 3. Предлагаемые endpoint-ы

Черновой контракт:

- `POST /api/v1/cases` — создать обезличенную сессию;
- `POST /api/v1/cases/{id}/answers` — передать ответы шага;
- `GET /api/v1/cases/{id}/next-question` — получить следующий обязательный вопрос;
- `POST /api/v1/cases/{id}/evaluate` — выполнить проверку и сформировать результат;
- `GET /api/v1/sources/{id}` — открыть метаданные источника;
- `POST /api/v1/feedback` — сообщить об ошибке или спорной рекомендации.

## 4. Формат клинического ответа

Backend возвращает строго структурированный объект:

```json
{
  "module": "dry-eye-v0.1",
  "algorithm_version": "0.1.0",
  "urgency": "routine",
  "red_flags": [],
  "phenotype": {
    "value": "evaporative_suspected",
    "confidence": "insufficient_for_clinical_use"
  },
  "missing_data": [],
  "differential": [],
  "investigations": [],
  "treatment_options": [],
  "sources": [],
  "limitations": [],
  "generated_at": "ISO-8601"
}
```

На этапе прототипа `treatment_options` всегда остаётся пустым.

## 5. Защита данных

Первая версия должна запрещать прямые идентификаторы пациента. Необходимо:

- предупреждать пользователя до ввода данных;
- автоматически выявлять возможные ФИО, телефоны, адреса, номера документов;
- отклонять или очищать подозрительные поля;
- не записывать свободный текст без необходимости;
- устанавливать срок хранения сессий;
- отделять технические логи от клинических данных.

## 6. Развёртывание

GitHub Pages остаётся витриной и frontend-слоем. Backend размещается отдельно и подключается по HTTPS через ограниченный CORS.

Возможная структура:

```text
matveyshemyakin.ru/doctors/          — интерфейс
api.matveyshemyakin.ru/              — backend API
admin.matveyshemyakin.ru/            — редакционная панель
```

Секреты хранятся только в серверных переменных окружения. API-ключ никогда не передаётся браузеру.

## 7. Критерии перехода к лекарственному модулю

Лекарственные варианты нельзя включать, пока не выполнены все условия:

1. утверждена структура лекарственной базы;
2. каждая доза привязана к первоисточнику;
3. реализованы противопоказания и блокировки;
4. создан набор клинических тестов;
5. проведено экспертное рецензирование;
6. определён регуляторный статус продукта;
7. утверждены юридические документы;
8. настроено версионирование и отзыв устаревших правил.
