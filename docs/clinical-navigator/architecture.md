# Архитектура клинического навигатора

## 1. Главный продуктовый принцип

Навигатор не хранит заранее написанный итоговый ответ и не принимает окончательное клиническое решение.

Система должна:

1. понять свободное описание врача;
2. извлечь уже сообщённые клинические факты;
3. не задавать повторно закрытые вопросы;
4. сформировать дифференциальный ряд;
5. найти релевантные фрагменты международных рекомендаций в контролируемой базе;
6. сгенерировать несколько диагностических и лечебных вариантов;
7. показать аргументы, риски, необходимые проверки и источники;
8. оставить окончательный выбор врачу.

```text
Клинический случай
        ↓
Извлечение фактов и память случая
        ↓
Выявление недостающих данных и красных флагов
        ↓
Поиск доказательных фрагментов по конкретному фенотипу
        ↓
Модель формирует несколько вариантов
        ↓
Safety validator проверяет ограничения и источники
        ↓
Врач сравнивает и выбирает тактику
```

## 2. Разделение ответственности

### Frontend

Отвечает за:

- интерфейс врача;
- ввод обезличенного случая;
- отображение извлечённых данных;
- уточнение только отсутствующих параметров;
- сравнение вариантов диагноза и лечения;
- фиксацию выбора врача;
- отображение источников и ограничений.

Frontend не хранит API-ключи и не выбирает лечение автоматически.

### Backend

Отвечает за:

- авторизацию и профессиональный доступ;
- очистку и структурирование входных данных;
- память обезличенной клинической сессии;
- вызов выбранной языковой модели через универсальный адаптер;
- поиск по библиотеке патологий и международных источников;
- выполнение жёстких правил безопасности;
- валидацию препаратов, дозировок, противопоказаний и мониторинга;
- журналирование версии пакета знаний и источников;
- лимиты, аудит и мониторинг ошибок.

### Языковая модель

Модель может:

- понимать свободный медицинский текст;
- извлекать признаки, отрицания, сроки и результаты обследований;
- строить и ранжировать дифференциальные варианты;
- формировать целевые поисковые запросы к базе знаний;
- сопоставлять найденные рекомендации с конкретным случаем;
- формировать несколько вариантов обследования и лечения;
- объяснять преимущества, риски и неопределённость каждого варианта.

Модель не может:

- выбирать окончательный диагноз или лечение за врача;
- придумывать препарат, дозу или длительность вне найденных источников;
- скрывать противоречащие факты;
- игнорировать красные флаги и блокировки;
- использовать открытый интернет в клиническом режиме;
- самостоятельно изменять библиотеку знаний.

### Rule engine и safety validator

Детерминированный слой определяет:

- обязательные данные для конкретной ветви;
- красные флаги и условия срочной маршрутизации;
- допустимые единицы и диапазоны;
- противопоказания и лекарственные взаимодействия;
- необходимость коррекции дозы;
- условия, при которых вариант нельзя показывать;
- правила мониторинга;
- запрет автоматического выбора варианта.

## 3. Расширяемая библиотека патологий

Каждая патология подключается отдельным версионируемым пакетом:

```text
backend/knowledge/pathologies/<pathology-id>/
├── manifest.json   — возможности, необходимые факты и контракт ответа
└── evidence.json   — атомарные доказательные фрагменты и ссылки на источники
```

Пакет содержит не готовый ответ, а:

- диагностические критерии;
- клинические фенотипы;
- дифференциальные признаки;
- варианты обследования;
- варианты лечения;
- условия применимости;
- риски и противопоказания;
- мониторинг и критерии эскалации;
- ссылки на конкретные международные источники;
- статус клинической проверки.

Добавление новой патологии не должно требовать изменения провайдера ИИ или переписывания интерфейса.

## 4. Статусы знаний

Каждый доказательный фрагмент получает один из статусов:

- `retrievable` — разрешён для формирования ответа;
- `draft_locked` — доступен только в закрытом редакторском режиме;
- `approved` — прошёл клиническую и юридическую проверку;
- `deprecated` — исключён из новых ответов.

Черновой лечебный материал не попадает в обычный пользовательский режим.

## 5. Контракт ответа

Модель должна вернуть структурированный набор вариантов:

```json
{
  "working_problem": "anterior_uveitis",
  "urgency": "prompt",
  "diagnostic_options": [
    {
      "option_id": "option-1",
      "diagnosis": "...",
      "supporting_facts": [],
      "contradicting_or_missing_facts": [],
      "suggested_investigations": [],
      "evidence_refs": []
    }
  ],
  "treatment_options": [
    {
      "option_id": "treatment-1",
      "clinical_intent": "...",
      "eligibility_conditions": [],
      "regimen_components": [],
      "benefits": [],
      "risks": [],
      "contraindication_checks": [],
      "monitoring": [],
      "failure_or_escalation_criteria": [],
      "evidence_refs": [],
      "evidence_strength": "...",
      "selected": false,
      "physician_selection_required": true
    }
  ],
  "sources": [],
  "limitations": []
}
```

Ни один вариант не может иметь `selected: true` до явного действия врача.

## 6. Текущие backend-действия

- `action: extract_facts` — извлечь и объединить факты из текста;
- `action: build_options` — получить доказательный контекст и контракт нескольких вариантов.

Для закрытой разработки лечебных пакетов используется серверный флаг `ALLOW_DRAFT_CLINICAL_OPTIONS=true`. В обычном режиме фрагменты `draft_locked` исключаются.

## 7. Защита данных

Система должна:

- запрещать прямые идентификаторы пациента;
- выявлять возможные ФИО, телефоны, адреса и номера документов;
- не хранить свободный текст дольше необходимого;
- отделять технические логи от клинических данных;
- фиксировать версию модели, пакета знаний и источников;
- позволять полностью удалить сессию.

## 8. Развёртывание

```text
matveyshemyakin.ru/doctors/          — frontend
api.matveyshemyakin.ru/              — защищённый backend
admin.matveyshemyakin.ru/            — редакционная и клиническая проверка
```

Провайдер модели выбирается конфигурацией backend: Yandex, OpenAI или OpenAI-совместимый API. API-ключи никогда не передаются браузеру.

## 9. Условия активации лечебного пакета

Лечебные варианты становятся доступными врачу только после выполнения всех условий:

1. каждый существенный вывод привязан к источнику;
2. каждая дозировка сверена с первоисточником и официальной инструкцией;
3. реализованы противопоказания, взаимодействия и коррекция дозы;
4. определены обязательные данные и условия блокировки;
5. создан набор позитивных, негативных и пограничных клинических тестов;
6. проведено независимое клиническое рецензирование;
7. определён регуляторный и юридический статус;
8. настроено версионирование, журналирование и отзыв устаревших рекомендаций.
